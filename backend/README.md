# PaperTrail Backend API
This is the backend for the PaperTrail digitisation pipeline, built with FastAPI, Firebase Admin, and Google Vision API.

## Setup Instructions

1.  **Environment Variables**:
    Copy `.env.example` to `.env` and fill in the necessary variables:
    *   `FIREBASE_SERVICE_ACCOUNT_PATH`: Ensure this points to `serviceAccountKey.json`.
    *   `FIREBASE_STORAGE_BUCKET`: E.g., `papertrail-e7ac6.appspot.com`
    *   `GOOGLE_VISION_ENABLED`: Set to `true` to use Google Vision, or `false` to fallback to local Tesseract OCR.
    *   `OCR_UNCERTAINTY_THRESHOLD`: Float, default `0.75`.
    *   `MATCH_CONFIDENCE_THRESHOLD`: Float, default `0.80`.

2.  **Dependencies**:
    Ensure you have Python 3 installed. Then run:
    ```bash
    pip install -r requirements.txt
    ```
    If you are using Tesseract (i.e. `GOOGLE_VISION_ENABLED=false`), ensure Tesseract is installed on your OS and added to your system PATH.

3.  **Firebase Credentials**:
    Ensure the `serviceAccountKey.json` is present in the `backend/` folder. This provides full administrative access to Firebase Auth, Firestore, and Storage.

4.  **Database Seeding**:
    We have provided a script to seed initial document templates. Run it once before starting the server:
    ```bash
    python seed/seed_templates.py
    ```

5.  **Running the Server**:
    Start the FastAPI application with Uvicorn:
    ```bash
    uvicorn main:app --reload --port 8000
    ```

## API Documentation
Once the server is running, interactive API documentation is automatically generated by FastAPI.
Visit: [http://localhost:8000/docs](http://localhost:8000/docs)

This page uses Swagger UI and includes the full OpenAPI specification, including all request/response schemas, available routes, authentication requirements, and error codes.

## Connecting from the Frontend
*   The API runs by default on `http://localhost:8000`.
*   **Authentication**: All protected routes require a Firebase ID token. When a user logs in via the Firebase Client SDK in Next.js, retrieve their ID token (`await user.getIdToken()`) and pass it in the `Authorization` header as a Bearer token:
    `Authorization: Bearer <ID_TOKEN>`
*   **Roles**: Ensure you configure custom claims or check the `users/{uid}` document in Firestore to verify if a user is `admin` or `clerk`. The backend relies on the custom claim or Firestore record structure created by the `POST /auth/register` endpoint.
