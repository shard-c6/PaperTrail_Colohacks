
API REFERENCE DOCUMENT
PaperTrail
Handwritten Government Form Digitisation Pipeline


1. Overview
This document is the authoritative API contract for PaperTrail. Both the backend team (building endpoints) and the frontend team (consuming them) must adhere strictly to the request and response shapes defined here. No field names, types, or status codes may be changed without a versioned update to this document.

1.1  Base URL

During development, the mock server runs at http://localhost:8000/v1. CORS is enabled for all origins in development; locked to the frontend domain in production.

1.2  Authentication
Every endpoint (except the Firebase-handled auth flows) requires a valid Firebase ID Token passed in the Authorization header:


The backend validates the token using Firebase Admin SDK, then reads the role field from Firestore (users/{uid}) to enforce role-based access. Roles: clerk and admin.

1.3  Standard error envelope
All error responses use the same JSON shape regardless of endpoint:


The code field is a machine-readable string the frontend uses to render the correct UI state. The message is human-readable. The details object carries additional context when available (e.g. which field failed validation).

1.4  Bounding box coordinate format
All bounding boxes throughout the API use relative coordinates — values between 0.0 and 1.0 representing fractions of the image's width or height. This allows the frontend to overlay boxes at any display resolution without recalculation.



2. Authentication
Authentication is handled entirely by Firebase Authentication on the client side. The backend never handles raw passwords. The flows below describe what the frontend must do — the backend only receives and validates the resulting Firebase ID token.

2.1  Sign-up
The frontend calls Firebase Auth's createUserWithEmailAndPassword(email, password) directly. On success, the frontend must immediately call POST /auth/register to create the user record in Firestore with the correct role.

Implementation notes
Role is always set to clerk on self-registration. Admin role can only be assigned by an existing admin via PATCH /admin/users/{uid}/role.

These endpoints map directly to Layers 1–6 of the backend pipeline. They must be called in order for a single form submission. Each response includes a session_id that links all layers of the same submission together.

Implementation notes
The frontend must display cleaned_image_url to the clerk for visual confirmation before calling /match-template.
session_id must be stored in frontend state and passed to every subsequent pipeline call.
original_image_url is stored permanently for the audit record and never modified.
If skew_corrected_degrees > 10, the frontend should warn the clerk that the image was heavily rotated and ask them to confirm the cleaned result looks correct.

Implementation notes
uncertain: true is set when confidence < 0.75. This threshold is configurable in backend environment variables (OCR_UNCERTAINTY_THRESHOLD).
